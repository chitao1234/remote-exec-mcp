# Phase D3 Operator Observability Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans to implement this plan task-by-task. If the user has expressed a strong preference for one of these execution styles, keep using that style unless they explicitly ask to switch. Steps use checkbox (`- [ ]`) syntax for tracking.
>
> **Plan rule:** This document is a merged design + execution artifact. Any code blocks are illustrative only. Concrete implementation code belongs in the actual code changes, not in this plan.

**Goal:** Improve operator debugging for broker and daemon failures by adding request correlation, secret-safe config debug output, graceful daemon shutdown, patch audit details, CLI exit-code taxonomy, and truthful Rust/C++ daemon config parity notes.

**Requirements:**
- Generate one broker tool-call correlation ID and propagate it to daemon HTTP requests using a shared request ID header.
- Include correlation ID, tool name, and target name where known in MCP tool error output and matching broker logs.
- Redact bearer tokens from `HttpAuthConfig` debug output.
- Add graceful SIGTERM/CTRL-C handling to the Rust daemon binary by using the existing shutdown-aware server entrypoint.
- Add daemon `base_url` to broker startup warning logs for unavailable or probe-timed-out targets.
- Improve patch auditability without changing public `apply_patch` text output.
- Clarify or close meaningful Rust/C++ daemon config drift, especially Rust exec session limits.
- Add CLI exit-code categories for usage/input, config, connection/transport, and tool errors.
- Explicit non-goal: do not include Log-Level Discipline from the D3 design in this plan. In particular, do not change `write_stdin` empty-poll log levels here.
- Explicit non-goal: do not add structured metrics in this plan. Metrics should be a separate D3b design and implementation slice.

**Architecture:** Add a lightweight request context at the broker tool boundary and pass it through broker tool execution and daemon client calls. Put shared wire constants in `remote-exec-proto`, keep the public MCP tool schemas unchanged, and use additive/defaulted internal RPC fields for patch audit data so older or reduced daemon paths remain decodable. Prefer explicit, small changes in each crate over a broad observability framework.

**Verification Strategy:** Use focused Rust unit/integration tests for request IDs, tool error formatting, patch response fields, CLI exit codes, config parsing, and daemon request header behavior. For C++ parity changes, use focused C++ route/config tests. Finish with targeted package tests for touched crates and the workspace quality gate when implementation is complete.

**Assumptions / Open Questions:**
- Request IDs should be generated by the broker per MCP tool call and reused for daemon calls made as part of that tool. If a tool performs multiple daemon calls, sharing the same request ID is intentional for end-to-end correlation.
- Direct daemon calls without a broker-supplied request ID should receive a daemon-local request ID for logging and response headers.
- Remote command process exit codes should not be mapped to the CLI process exit code in this plan unless current tests or docs already require that behavior. D3 only classifies CLI/tool/config/connection failures.
- C++ daemon changes should be included where they share the same operator-visible contract, especially `x-request-id` logging and patch audit response shape.
